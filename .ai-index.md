### 🛡️ Rule Enforcement Mechanisms

Every commit/PR to this framework is validated through **4-layer enforcement:**

**Layer 1: Pre-Commit Hooks** (Local)
- Run automatically when you `git commit`
- Validate ia-rules compliance
- Block commits with violations
- Check: `.sdd-core/spec/CANONICAL/rules/ENFORCEMENT_RULES.md`

**Layer 2: CI/CD Pipeline** (GitHub Actions)
- Runs architecture compliance tests
- Validates definition-of-done checklist
- Validates checkpoint updates
- Blocks PRs that fail

**Layer 3: Code Review** (Human)
- Reviewer checks thread isolation (no conflicts)
- Reviewer validates checkpoint updates
- Reviewer verifies test coverage
- PR blocks merge until approved

**Layer 4: Continuous Metrics** (Audits)
- Weekly rule violation audits
- Framework compliance reports
- Quality metrics: `context/runtime-state/metrics/`

**You must ensure:**
- ✅ Pre-commit passes (`git commit` succeeds)
- ✅ All CI/CD checks green (before PR)
- ✅ Checkpoint documented (in PR description)
- ✅ Thread isolation respected (check execution-state/)

## 🔑 Key Concepts You Must Understand

### Ports & Adapters Pattern (ADR-003)
Your code should be organized like this:

```
src/[domain]/
├── ports/                    (interfaces)
│   ├── storage_port.py       (what storage should do)
│   └── notification_port.py  (what notifications should do)
├── use_cases/               (business logic)
│   └── create_thing.py       (uses ports, not implementations)
└── adapters/                (implementations)
    ├── postgres_storage.py   (implements StoragePort)
    └── email_notification.py (implements NotificationPort)
```

**Rule:** Import ports from `src.domain.ports`, never from `src.infrastructure`

### Definition of Done (45+ Criteria)
File: `.sdd-core/spec/CANONICAL/specifications/definition-of-done.md`

Your feature isn't done until you check all 45+ items:
- ✅ Tests passing (100% coverage for new code)
- ✅ Documentation updated
- ✅ No rule violations
- ✅ Architecture validated
- ✅ (40+ more items)

### Thread Isolation
Each person/agent works on ONE task at a time.

- Check: `.sdd-core/spec/custom/[YOUR_PROJECT]/development/execution-state/_current.md`
- If someone else is working on your task: Wait or coordinate
- After you're done: Update checkpoint with what you did

### Checkpointing
After finishing work, document it:

```markdown
# Checkpoint Entry [TIMESTAMP @ AGENT_NAME]

**TASK:** What did you implement?
**DECISIONS:** What choices did you make?
**QUESTIONS:** What's unclear?
**RISKS:** What could go wrong?
**STATUS:** Ready for PR
```

This helps the next person know what you did.
