# Git Hooks Architecture

**Two separate git hook systems: Framework vs Client**

---

## Overview

The SDD project uses two distinct git hook strategies:

```
┌─────────────────────────────────────────────────────────────┐
│ SDD Architecture Framework                                  │
│ (sdd-architecture repo)                                     │
│                                                             │
│ .git/hooks/pre-push                                         │
│ ├─ Validates: Health (10 checks)                           │
│ ├─ Validates: Governance (4 mandates, 151 guidelines)      │
│ ├─ Blocks merge if: Compliance < 100%                      │
│ └─ Purpose: Protect framework integrity                    │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│ Client Projects                                             │
│ (Generated by wizard)                                       │
│                                                             │
│ .git/hooks/pre-push (Optional seedling)                     │
│ ├─ Validates: Project tests pass                           │
│ ├─ Validates: Code lint OK                                 │
│ ├─ Validates: Project health checks                        │
│ ├─ Does NOT validate: SDD governance                       │
│ └─ Purpose: Protect project quality only                   │
└─────────────────────────────────────────────────────────────┘
```

---

## Framework Hooks (SDD Architecture)

### Location
`.git/hooks/pre-push` in sdd-architecture repository

### Responsibility
```bash
✅ Validates: Health check (10 system checks)
   ├─ Git repository status
   ├─ Core structure (5 subsystems)
   ├─ Docs structure (6 subdirectories)
   ├─ Python version (3.8+)
   └─ ... (10 total checks)

✅ Validates: Governance compliance
   ├─ 4 immutable mandates
   ├─ 151 customizable guidelines
   ├─ Enforcement level (STRICT/STANDARD/PERMISSIVE)
   └─ Compliance score must = 100%

✅ Blocks merge if: Any validation fails
```

### Behavior
```bash
$ git push origin main

[PRE-PUSH] Starting final validation before push...
→ Running full health check (fresh, no cache)...
✓ Health check passed (10/10)
→ Verifying governance enforcement level...
✓ Enforcement mode: STRICT
→ Checking compliance percentage...
✓ Fully compliant (100%)
[PRE-PUSH] All checks passed. Push allowed.
```

### Skip (if absolutely necessary)
```bash
git push --no-verify
```

## Client Project Hooks (Generated Projects)

### Location
`.git/hooks/pre-push` in client project (optional)

**Note:** Currently TBD - not yet implemented in wizard templates

### Responsibility (Planned)
```bash
✅ Validates: Project tests pass
   ├─ Runs pytest or project test command
   └─ Ensures 100% test pass rate

✅ Validates: Code quality (lint)
   ├─ Runs pylint, mypy, black
   ├─ Checks code formatting
   └─ Type hints validation

✅ Validates: Project health
   ├─ Dependencies available
   ├─ Project structure OK
   └─ Build configuration valid

❌ Does NOT validate:
   ├─ SDD Architecture governance
   ├─ Framework mandates
   ├─ Framework guidelines
   └─ Framework compliance
```

### Behavior (Planned)
```bash
$ git push origin main

[PRE-PUSH] Running project validation...
→ Running tests...
✓ All tests passed (45/45)
→ Running lint checks...
✓ Code quality OK
→ Checking project structure...
✓ Project healthy
[PRE-PUSH] All checks passed. Push allowed.

# Note: Does NOT check SDD governance
```

### Why Separate?

**Framework hooks validate framework rules** because:
- Framework repo must stay governance-compliant
- Protects integrity of wizard, compiler, health checks
- Enforces 4 immutable mandates
- Ensures 151 guidelines are respected

**Client hooks validate project rules** because:
- Each project has its own quality standards
- Clients may NOT adopt SDD governance
- May use different testing frameworks
- May have different linting rules
- Should NOT be forced to follow SDD mandates
